🎬 SyncWatch Bridge Setup

The bridge userscript lets SyncWatch control the video player inside the embed page. Everyone in the room needs to install it once.

Step 1 — Install a userscript manager

Tampermonkey (Chrome / Edge)  Violentmonkey (Firefox)

Either works. Install the extension and make sure it's enabled.

Step 2 — Install the SyncWatch Bridge script

Your userscript manager will open a tab and ask you to confirm. Click Install there.

⚠️ The install tab may close itself immediately after you click Install/Update. This is normal Tampermonkey behavior — it's not a crash, and it doesn't mean the install failed. The tab's only job was to show the install prompt; once you confirm, Tampermonkey is done with it and closes it automatically. Use the checker below to actually confirm whether it worked instead of relying on the tab staying open.

✓ Verify it actually installed

Since the install tab disappears too fast to read, click below to check directly.

Step 3 — Reload the room page

Go back to your SyncWatch room and reload. The yellow warning bar should disappear once you load a stream.

Supported embed sites (built in)

The bridge currently works out of the box on:

streamimdb.ru, playimdb.com, vidsrc.to, vidsrc.me, vidsrc.cc, vidsrc.in, embed.su, multiembed.mov, superembed.stream, 2embed.cc, 2embed.to, moviesapi.club, player.vimeo.com.

📍 Using a different site? Add it yourself — takes 30 seconds

Embed aggregator sites change domains constantly (that's exactly why streamimdb.ru became playimdb.com at one point). If your movie's domain isn't in the list above, the bridge script genuinely never runs on that page at all — this isn't a bug, Tampermonkey only activates scripts on domains they explicitly list. Here's how to add one yourself:

  1. Click the Tampermonkey icon in your toolbar → Dashboard
  2. Find SyncWatch Bridge in the list, click its name to open the editor
  3. Near the top, find the lines that start with // @match
  4. Add a new line just like the others, using the website's actual domain. For example, if the room's video is loading from https://playimdb.com/title/tt123/, add:

    // @match *://playimdb.com/* // @match *://*.playimdb.com/*
  5. Press Ctrl + S to save
  6. Go back to your SyncWatch room and reload the page — the bridge will now run on that domain

Tip: the exact domain to use is whatever shows in your browser's address bar when you look at just the embed iframe — right-click the video area → "This Frame" → "Open Frame in New Tab" to see it directly if you're not sure.

⚠️ Bridge says "detected" but play/pause still doesn't work?

This is a different, more specific problem from the one above — and it's actually very common. Many aggregator sites (playimdb.com included) don't put the actual video player on their own page at all. Instead, their page embeds a second, nested iframe from a totally separate video-hosting domain, and the real <video> element lives inside that inner frame — not on the page whose URL you pasted.

The bridge (v2.0+) automatically relays commands down into nested iframes, but only into ones whose domain is also in the @match list — it can't inject itself into a domain it's never heard of, same as before. If a site's actual player lives on a CDN domain we don't already cover (we include the most common ones: rabbitstream.net, streamtape.com, streamwish.com, filemoon.*, doodstream.com, mixdrop.*, and a few others), here's how to find and add it:

  1. Open the embed URL directly in a normal browser tab (not inside SyncWatch)
  2. Right-click directly on the video itself → look for "This Frame" in the context menu
  3. Hover "This Frame" → click "Open Frame in New Tab" (or similar, varies by browser)
  4. Check the address bar of the tab that opens — that's the real player domain
  5. Add it to the bridge's @match list the same way as Step 4 above

If right-click is disabled on the page (some sites block it), open DevTools (F12) → Elements tab → look for a nested <iframe src="..."> tag and read its src domain directly from there instead.

🚫 Screen shows "refused to connect" instead of the movie?

This is a different problem from everything above — and unlike those, this one genuinely can't be fixed by installing or configuring anything. If the player area shows Chrome's own error page with wording like "localhost refused to connect" rather than any part of the actual movie site, the site you loaded is sending a security instruction called X-Frame-Options (or an equivalent Content-Security-Policy rule) that tells every browser, everywhere: "never display me inside anyone else's page."

This is the exact same protection that stops a malicious site from secretly embedding your bank's login page to steal clicks — and browsers enforce it unconditionally. No setting in SyncWatch, no Tampermonkey config, and no @match addition can override it, because the block happens at the browser level before any of our code even runs.

The fix: use a different mirror or source for the same movie. Many aggregator sites mirror the exact same content under several different domains specifically because some allow embedding and some don't — if one fails this way, try another. This is also why testing the URL by itself in a normal tab first is useful: if it opens fine there but fails inside SyncWatch with this exact error, you've confirmed it's this issue and not anything else on this page.

Honest limitation: some sites can't be fixed this way

Occasionally a nested player domain changes on every single page load (some sites do this deliberately to make ad-blocking and exactly this kind of automation harder), in which case no fixed @match list can ever keep up. If a site behaves this way after you've correctly identified and added its current player domain, it's a strong sign that specific site isn't a good fit for SyncWatch — worth trying a different mirror or source for the same movie instead of fighting that one site further.

Still not detecting after install?

1. Open Tampermonkey's icon in your toolbar → Dashboard. You should see "SyncWatch Bridge" in the list, enabled (toggle switch on, not greyed out).

2. If it's NOT there: the install genuinely didn't save. Try again, and this time watch Tampermonkey's toolbar icon — it usually shows a small "+" badge for a second when a new script is detected, even if the tab itself closes fast.

3. If it IS there but still not detected in a room: open the embed site directly in a tab (e.g. paste a vidsrc.to URL straight into your address bar) and check the browser console (F12) for a line starting with [SyncWatch Bridge] Loaded on. If you don't see it, the site's @match pattern doesn't cover that exact domain — see the section above to add it yourself.
